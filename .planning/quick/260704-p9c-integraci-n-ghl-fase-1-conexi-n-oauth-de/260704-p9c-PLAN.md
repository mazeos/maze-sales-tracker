**Helpers de state firmado (node:crypto):**

- `verifyState(state)` → separa por `.`, recalcula la firma y compara con `crypto.timingSafeEqual` (sobre Buffers de igual largo; si difieren de largo → inválido), parsea el payload y exige `Date.now() - ts < 10 * 60 * 1000`. Retorna `{ok:true, org}` o `{ok:false}`.

**`GET /api/oauth/start`** — es una navegación del browser (no fetch), así que el JWT llega por query `?token=`. **`GET /api/oauth/callback?code=…&state=…`** — sin auth de sesión (viene de GHL); la autenticidad la da el state firmado:

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → mini-API | JWT del usuario cruza por header y (solo en /oauth/start) por query param |
| GHL → /api/oauth/callback | Request externo sin sesión; cualquiera puede llamar la URL |
| mini-API → PostgREST | Service role bypassea RLS; el filtrado por org es manual |
| Repo público → mundo | Cualquier secreto commiteado queda expuesto |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-q-01 | Spoofing | /api/oauth/callback | mitigate | state HMAC-SHA256 (SERVICE_ROLE_KEY como secreto) + timingSafeEqual + expiración 10 min; state inválido → redirect con ghl_error, sin tocar DB |
| T-q-02 | Tampering | state (org_id embebido) | mitigate | org_id viene del JWT validado en /oauth/start y va firmado; el callback usa SOLO el org del state verificado |
| T-q-03 | Information Disclosure | tokens OAuth en st_integrations | mitigate | RLS habilitada SIN policies ni grants (deny-all vía PostgREST); GET /api/integrations/ghl hace select explícito sin columnas de tokens |
| T-q-04 | Information Disclosure | GHL_CLIENT_SECRET en repo público | mitigate | secretos solo por env vars; nunca en código/plan/commits (valores reales en Fate Vault, los coloca el orquestador en el .env del VPS) |
| T-q-05 | Information Disclosure | JWT en query param de /oauth/start | accept | JWT de vida corta, TLS end-to-end, uso único para una navegación; alternativa (cookie/POST) rompe el patrón sin-deps. Riesgo residual: logs de nginx — aceptado para fase 1 |
| T-q-06 | Elevation of Privilege | /api/oauth/start y /api/integrations/ghl | mitigate | misma validación checkAdminToken que requireAdmin (GoTrue + role=admin en st_profiles); no-admin → 403 |
| T-q-07 | Denial of Service | callback con codes basura | accept | el exchange falla rápido contra GHL y redirige; sin estado persistente creado antes de validar. Volumen esperado ínfimo |
| T-q-SC | Tampering | supply chain npm | accept | cero dependencias npm nuevas (regla del proyecto: solo módulos nativos de Node) |
</threat_model>
